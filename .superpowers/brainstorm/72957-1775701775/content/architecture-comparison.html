TextInput 架构对比 (真相来源)

决定 WASM (Guest) 与 Native (Host) 之间的同步方式

A

Guest 中心化 (受控组件)

WASM 持有主状态。 Native 每次按键都发给 Guest，Guest 过滤后返回给 Native 更新输入缓冲区。

  • 极致的控制力 (如输入掩码、撤销逻辑)
  • 与 Xilem/React 逻辑完美契合
  • FFI 延迟可能导致 IME 候选词抖动
  • 处理亚洲语言 (IME) 极其复杂
B

Host 中心化 (原生驱动)

Native 持有主状态。 使用平台原生的 TextInputView，WASM 仅接收“文本已变更”事件并渲染结果。

  • IME 体验最流畅 (iOS/Android 原生体验)
  • 开发复杂度最低
  • WASM 端很难实时拦截/修改输入
  • 渲染可能与原生输入框有短暂不同步
C

混合/事务型 (解耦合)

Flutter 模式。 Native 通过 TextInputClient 与 WASM 通信，变更以事务形式同步。WASM 端维护对应的 TextEditingController。

  • 兼顾流畅度与逻辑控制
  • 更易扩展到多行、富文本等复杂场景
  • 实现逻辑最复杂
  • 需要精细处理状态调和 (Reconciliation)
