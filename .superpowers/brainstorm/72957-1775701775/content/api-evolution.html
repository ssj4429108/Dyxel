API 进化：响应式值 (Reactive Value)

平衡简洁性 (A) 与 控制力 (B)

A

简单信号 (Signal<String>)

只关注内容。丢失光标、选区、IME 组合状态。

  • 无法处理中文输入法(IME)中间态
  • 无法实现代码高亮
精

响应式值 (Signal<TextState>)

推荐方案。 将文本、光标、选区打包进一个响应式对象。保留 A 的声明式语法，获得 B 的控制力。

let state = create_signal(TextState::new("Hello"));

TextInput {
  value: state,
  on_change: |s| state.set(s),
}
  • FFI 频率与 A 一致
  • 支持 IME 组合状态显示
  • 支持外部控制光标
B

重型控制器 (Controller)

完全命令式。手动管理事务、订阅者和 FFI 指令。

  • 容易导致内存泄漏 (循环引用)
  • 不符合 Xilem 的纯函数式视图理念
数据流：为什么 FFI 开销是一样的？

Native 输入 "A" → [FFI Call] → WASM 接收

Payload 差异：

  • 方案 A: "A" (1 byte)
  • 响应式值: { text: "A", cursor: 1 } (9 bytes)

结论：FFI 的开销主要在“调用”动作本身，Payload 大小微增对性能无感。
